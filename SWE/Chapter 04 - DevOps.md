# DevOps
- 애자일이 지원하지 못하는 문제를 토론하면서 등장
## DevOps 소개

- 사일로 기반 개발 방식
	- 사일로 개발 방식이란
		- 곡물 창고가 줄지어 서 있고, 하나의 창고에 곡물이 가득 쌓이면 다음 창고로 옮겨 쌓는 형태를 소프트웨어 개발에 대입 한 것
	- 개발의 각 단계가 완료되어야 다음 단계로 진행 할 수 있으며 각 단계 간에는 특별한 상호작용이 존재하진 않다.
	- 각 부서에 부여된 태스크를 충실히 수행하는 데만 관심이 있음
		- 다른 부서와 정보 공유 및 협업을 하는데는 적극적이지 못함
- 사일로 기반 개발 방식의 문제점
	- 팀 중복 작업 : 팀 간에 동일한 활동이 중복적으로 발생한다.
	- 고객 니즈 반영의 어려움: 시스탬 개발후 새로운 니즈가 출현하는 경우
	- 정보 활용 적시성 미흡: 다른 팀이 사용하는 장비와 정보를 찾기위해 노력과 시간이 소모죔
	- 팀 간 경쟁과 협업 미흡: 팀의 실저 향상가ㅗ 성과 달성에 도움이 된 아이디어가 공유되지 않아 다른 팀에서 같은 작업을 반복하는 경우
	- 위험한 합의 : 의견 불일치가 생겼을 때 재작업이 필요함에도 불구하고 결정된 사안을 변경하려고 하지 않음
	- 생산성 저하: 중복작업, 재작업, 어려운 정보 수집, 산출물에 대한 서로 다른 품질 수준등으로 인해 생산성이 심각하게 저하
	- 제품 품질의 저하: 의사소통과 협업이 미흡하기 때문에 아이디어가 공유가 안돼서 좋은 솔루션을 놓팀

- 팀마다 고유한 전문분야와 주어진 역활을 가지고 모든 것을 투명하게 공개하거나 공유할 필요가 없다.
	- 하지만 본인의 작업에 몰두하게 되면서 팀 간 협업과 의사소통이 되지 않음
### DevOps의 정의
- 애자일에 뿌리를 두고 있다
	- 신속하게 소프트웨어 모듈을 개발하고 사용자에게 배포하는 것
- 차이점
	- 배포된 소프트웨어에 대한 사용자 모니터링
	- 사용자 모니터링을 통해 배포된 소프트웨어의 문제점과 개선점을 신속하게 찾는다.
	- 이를 소프트웨어에 즉시 반영한다.
- 정의
	- 개발, 운영, 품질 보증 및 서비스 제공자가 상호 협력하여 소프트웨어 시스템을 개발, 배포하고 시장 기회의 포착과 고객 피드백을 신속하게 처리함으로 서비스를 적시에 제공할 수 있도록 한다.

## 린 사고방식과 원리
-  DevOps에서는 린 사고방식과 원리를 반영한다.

- 린사고 방식이란
	- 낭비 요소를 제거하면서 사회에 더많은 해택을 제공하고 개인이 일을 통한 가치를 얻을 수 있도록 인간 활동을 체계화하는 방법을 제공하는 것을 목표로 둔 접근 방법 이다.
- 린 사고의 목표
	- 고개만족과 직원 만족도를 일치시켜 성장을 지속하고 고객, 공급업체및 환경에 불필요한 초과 비용을 최소화하면서 수익성 있는 혁신적인 제품 또는 서비스를 제공하는 것
	- 낭비되는 시간 식별 => 낭비되는 시간 제거 => 프로세스 개선 => 더 잘 협력 => 적은 비용으로 큰 가치를 생성

- 린 소프트웨어
	- 린 제조 원칙과 관행을 소프트웨어 개발 영역에  적용하는 것
	- 크게 생각하고, 작게 행동하고, 빨리 실패하고, 빠르게 배워라

### 린 소프트웨어 7가지 원칙
#### 낭비를 제거하라
- 제품 개발시 발생하는 낭비 요소를 식별하고 삭제한다
	- 사용하지 않는 문서 작업, 고객이 사용하지 않는 기능, 팀원간 작업 적환, 다른 작업을 기다리는 시간
- 철저한 스케줄 계획과 작업 할당, 프로세스에 대한 재구성등이 필요할 수 있다.
#### 학습을 확대하라
- 분석 및 설계 활동, 코딩 활동 등을 반복적으로 수행하면서 효율적으로 수행할 수 있는 방법을 찾는다.
- 상용자 영역 및 도메인에 대한 학습을 수행한다.
- 학습은 짧은 주기에 신속하게 이루어지도록 해야한다.
#### 최대한 늦게 결정하라
- 급변하는 니즈를 반영하기 위해서 최대한 결정을 미루는 것이 매우 효과적일 수 있다.
- 소프트웨어 개발은 불완정성을 안고 진행하기 때문에 가정인 ㅏ에측에 의한 의사결정보다 사실에 기반하여 의사 결정을 할 수 있도록 최대한 미루는 것이 유리하다.
#### 최대한 빨리 배포하라
- 신속하게 개발해서 사용자에게 전달하는 하는 것
- 설계, 구현, 피드뱍, 개선이라는 사이클을 신속하게 진행하여 어제의 요구사항이 아닌 현재의 요구사항을 반영해야 한다.
- 매일 주기적으로 미팅하는 것도 신속성을 향상할 방법이 될 수 있다.
#### 팀에 권한을 위임하라
- 전문적인 작업은 수행할 때는 해당 상황을 잘 알고 있는 것 담당자이다.
- 관련사항은 관련된 팀이나 팀원에게 위임한다
	- 팀장은 어려운 상황에서 지원과 도움을 제공한다.
#### 통합 지원 체계를 구축하라
- 통합은 빈번히 발생한다
- 따라서 구성품에 대한 표준화, 재사용 가능한 적절한 크기의 모듈 구성, 쉽게 진행할 수 있는 통합 등이 가능한 환경을 구축해야한다.
- 시스템에 대한 유연한 아키텍처 및 재사용과 확장성이 좋은 소프트웨어를 개발함으로써 가능해진다.
### 전체를 최적화 하라
- 자신의 팀 결과물의 우수성 보다 전체 시스템 최적화에 초점을 맞추어야 한다

## DevOps 표준: IEEE Std 2675
- 소프트웨어 수명 주기 프로세스를 정의하고 제어하고 개선하기 위한 목적으로 DevOps 구현에 대한 요구사항과 가이드라인을 제공

- 주요 내용
	1. 표준의 범위와 목적
	2. 참고 문헌
	3. 용어 및 약어 정의
	4. 적합성: 프로세스, 산출물, 태스크 등에 대한 표준 정합성의 설명
	5. DevOps의 개념: 가치, 원리, 조직, 문화, 수명주기 프로세스
	6. DevOps와 소프트웨어 수명주기 프로셋의 관계를 목적, 태스크, 산출물 관점에서 설명
		- 소프트웨어 수명주기 프로세스와의 관계에서 4개의 프로세스 그룹에 대한 세부 프로세스를 정의
			- 각 프로세스의 목적, 기존의 프로세스 표준, 주요 산출물, 해당 프로세스에서 수행하는 활동
## DevOps 성공 전략
- 조직 내 모든 기능 간의 협력 및 협업 문화가 조성되어야 한다
- 소프트웨어 개발의 근보적인 변화가 요구된다
	- 기존: 분석, 설계, 구현과 같은 독립된 역활 중심 활동
		- 정보 공유가 단순히 입력과 출력이라는 단일 경로로만 이루어져 있다

### DevOps 필요성 평가
- 단순히 유행엥 편승하기 위해서 도입을 고려하는 것은 비효율적임
	- 조직 문화 변화
	- 프로젝트 프로세스 변화
- 비지니스 차원에서 수립된 목표를 달성하기 위하여 DevOps 도입 목표가 정렬되어야 한다.

### 기존 사일로 방식의 타파
- DevOps의 철학은 본질적으로 개발, 운영 및 기타 기능이 서로 협력하여 긴밀하게 작업을 수행하는 것
- 기존 기능 및 역할 중심에서 팀 간 의사소통을 향상
- 과거에 수행한 작업, 관련된 사람, 작업 결과에 신속하게 접근할 수 있어야 한다.
	- 더 좋은 결과와 더 좋은 아이디어를 얻는 방법
### 고객 중심의 만족도 향상
- 끊임없이 변화하는 고개 요구에 계속 적응해야 하면 시간, 기능 및 성능에 대한 고잭의 기대치를 넘어서는 서비스 및 솔루션을 제공해야 한다.
	- 이해관계자의 참여
	- 팀의 노력
	- 투명한 의사소통
	- 고객의 기대티에 대한 지속적인 모니터링

### 작은 시작과 점진적 확장
- 작은 부분부터 변화를 유도해야 한다
	- 조직 문화를 바꾸어야 함으로 
	- DevOps 기반의 시스템 개발에서도 작은 릴리즈 주기 설정을 통한 신속한 개발과 통합 과정이 이루어 질수 있도록 시도
- 작은 변화를 기반으로 DevOps의 점진적인 확대를 도모

### 가능한 모든 곳의 자동화
- 신속한 소프트웨어 수명 주기 단게를 지원히기 위해서는 단절 없는 자동화가 필요
- 코드 개발, 통합을 위한 미들웨어, 자동 형상 구성 및 배포, 회귀 테스트 및 부화 테스트 등의 영역에서 필수적으로 이루어져야 한다.
- 개발자, 테스터, 운영자의 시간과 노력을 절약할 수 있어야 한다.
### 호환성이 보장되는 도구
- 자동화 도구는 다른 도구와의 호환성 및 상호 운용성이 제공되어야 한다.
- 기존 환경과 호환되는 도구 체인을 선택함으로써 도구간의 밀접한 통합
- 개발과 운영간의 발생할 수 있는 충돌 현상을 제거해야 한다
### 개인과 함께 팀 성과 검토
- DevOps 환경에서는 상호 협력과 협업을 총한 시스템 개발 및 운영이 이루어 진다
- 개인의 성과 뿐 아니라 팀의 성과도 평가되어야 한다.
- 개발 및 배포에 대한 목표를 달성할 수 있는 팀의 능력을 기반으로 평가해야 한다.
### 프로젝트 정보의 실시간 가시성 확보
- 다수의 팀에서 진행하는 프로젝트 상태 정보에 대한 실시간 접근이 중요
- 모든 이해관계자들이 현재 진행되는ㄴ 단계가 전체 과정의 어느 위치에 있으며, 누가 어떤 자원을 사용하고 있는지 확인할 수 있어야 한다
- 진보된 프로젝트 관리 도구는 이러한 프로젝트에 대한 포트폴리오 정보를 제공한다
### 지속적 통합 및 배포
- 지속적 통합및 지속적 배포가 구현되지 않았다면 DevOps를 구현했다고 볼 수 없다.
- 개발자가 결함을 즉시 감지하고 기민하게 소프트웨러를 개발하여 통합 과정으로 전달 해야 한다.
	- 지속적 통합
- 모든 변경 사항이 엄격하게 자동화된 테스트가 과정을 거쳐 사용자 시스템에 배치되어야 한다.
	- 지속적 배포
- 지속적 통합과 배포 프로세스와 이의 자동화가 핵심 요소가 된다.
### 모니터링 및 피드백을 통한 개선
- 소프트웨어 시스템 혹은 서비스가 최적의 수준에서 정확히 동작하고 있는지 지속적으로 모니터링 해야한다
- 모니터링 결과 발견된 문제저밍나 병목 현상에서 신속하게 대응해야한다.
- 사용자 피드백을 활용하여 지속적으로 시스템을 개선해야 한다
- 운영 과정에서 모니터링은 피드백을 즉각적으로 개발팀으로 전달하는 프로세스 개선이 이루어 져야한다.
## DevOps 프로세스
### 계획 단계
- 애플리케이션에 대한 비즈니스 가치와 사용자 요구사항 명확히 정의
- 대부분의 IT 부서 관계자들이 참여
	- 애플리케이션 소유자
	- 개발자
	- 설계자
	- 배포 담당자
	- 보안 담당자
	- 인프라 관리자
- 주요 활동
	- 개발 척도 및 대상 선정하고 피드백 검토
	- 요구사항 명세
	- 비즈니스 지표 정렬
	- 배포 척도 조정
	- 배포 계획, 시기, 비즈니스 사례를 정의
	- 보안 정책및 보안 요구사항을 정의
- 과정
	- 코딩 -> 단위 테스트 -> 커밋 -> 빌드 & 통합 테스트 -> 인수테스트, 성능 테스트 -> 배포 및 설치

### 코딩 단계
- 요구사항을 기반으로 소스코드를 개발
- 세부 활동
	- 소프트웨어 설계, 알고리즘 개발
	- 제품 형상을 설계하고 정의
	- 코드 품질과 개발 생상성을 고려하여 소스코드 작성
	- 개발자 입장에서 수행하는 단위 테스트 수행
### 빌드 단계
- 소스코드를 통합하고 소프트웨어 제품 형상을 개발하는 과정
- 기존 소프트웨어와 완전하게 동합해야 한다

- 세부 활동
	- 코드 통합 및 통합 테스트 수행
	- 제품에 대한 형상 구성
	- 배포를 위한 개발 애플리케이션을 준비
	- 애플리케이션 버전을 관리한다
### 테스트 단계
- 소프트웨어 품질을 유지하고 최상의 품질을 배포할 수  있도록 테스트, 검증해야한다.
- 자동화 도구 사용, 정적 분석 등의 방법으로 수행

- 세부 사항
	- 요구사항의 만족도를 테스트
	- 변경, 개선 등이 포함되는 활동에서 회귀 테스트를 수행한다
	- 애플리케이션에 대한 보안 및 취약성 분석을 수행한다
	- 성능을 테스트한다.
	- 소프트웨어 형상을 테스트 한다.
### 릴리즈 단계
- 크게 패키징과 배포로 구분되어 수행

- 세부 사항
	- 배포를 위한 승인 을 수행
	- 명세된 형상에 따라 패키징 한다
	- 배포를 위한 조정 과정을 수행
	- 수립된 배포 계획에 따라 애플리케이션을 배포한다
	- 버전 관리 정보를 기반으로 배포
### 설치(배포) 단계
- 사용자의 환경에 맞도록 애플리케이션을 서비스 플랫폼에 설치하는 과정
- 종종 배포 단계와 통합되어 수행되는 경우가 많다.

- 세부 활동
	- 배포된 애플리케이션을 사용자 환경에 설치
	- 필요시 스토리지, 데이터베이스, 네트워크등의 인프라를 구축한다.
	- 설치 과정에서 발생한 문제를 복구한다
	- 설치된 애플리케이션을 사용자에게 오픈한다

### 운영 및 모니터링 단계
- 설치된 애플리케이션을 기반으로 사용자에게 서비스를 제공
- 운영 과정에서 문제가 발생하는지 지속적으로 모니터링을 수행한다

- 세부사항
	- 인프라 구성 요소의 성능을 지속적으로 모니터링한다
	- 사용자의 반응 및 경험을 추적, 모니터링한다
	- 개발 척도의 달성 여부를 확인 한다.
	- 모니터링 결과를 지속적으로 개발 팀에 제공한다.
## DevOps ToolChain
- DevOps 프로세스를 실현하기 위한 적절한 자동화 도구를 선정해야 한다.
- 서로 다른 단계를 부분적으로 지원
- DevOps의 주요 목표중 하나는 애플리케이션 배포 및 운영 환경과 개발 및 테스트 환경 간 격차를 줄임으로써 애플리케이션 배포 시점에서 발생할 수 있는 환경 불일치 문제를 완화하는 것이다.
	- 이러한 도구들의 모임을 DevOps ToolChain이라고 한다
- 구성을 위한 정해진 규칙은 없지만 DevOps의 원칙에 따라 도구들이 잘 연동되고 통합 될 수 있는지 를 황ㄴ해야한다.
	- 성공적인 DevOps프로젝트의 Toolchain 구성은 벤치마킹하는 것도 좋은 방법이다.
	- 작은 규모의 프로젝트라면 도구들은 3~4개만 활용해도 되지만 큰 규모의 프로젝트라면 10개 이상의 도구가 필요할 수 도 있다.
- 최소한의 버전 제어도구, 자동화 도구 패키지 관리자, 테스트 도구가 포함될 수 있다.
## DevOps와 클라우드
- 클라우드 컴퓨팅은 컴퓨터 시스템의 자원을 사용자의 특별한 관리 없이 필요에 따라 사용할수 있도록 지원하는 플랫폼의 일종
	- 가상화를 토한 사용자 서비스 제공
	- 컴퓨터를 구매하지 않고도 편리하게 언제 어디서나 접곳하여 필요한 작업을 수행할 수 있다.
- IT 조직의 관점에서 유연성, 평의 성, 민첩성을 증진하면서 비용을 절감할 수 있다는 장점이 있다
- 클라우드 플랫폼엣 DevOps를 구현한다면 ㅂ매우 편리하고 간소하게 서비스 개발 및 배포가 가능해져 매우 유리할 수 있다.
### DevOps를 위한 크라우드 활용
- 설치, 운영될 플랫폼 환경과 개발 환경 간의 미묘한 차이로 인하여 설치 오류가 많인 발생한다.
	- 클라우드 플랫폼상에 운영 환경과 동일한 플랫폼을 구축하고 서비스를 개발해야 한다.

- 장점
	- 운영 환경을 신속하게 구축할 수 있기 때문에 On-Demand 시점의 가용성 및 접근성을 제공받을 수 있다.
	- 다양한 테스트를 쉽게 수행할 수 있다
	- 서로 다른 환경에서 테스트나 테스트 변경에 민첩하게 대응할 수 있다.
	- 환경 구축, 해체가 용이하여 비용이 절감된다.
	- 개발자 혹은 테스터가 요구하는 환경을 패턴으로 정의하고 패턴을 이용하여 실무자가 요구하는 환경을 one-click으로 제공 받을 수 있다.
	- DevOps를 위한 Toolchain을 모두 구매하지 않고 On-Demend 방식으로 편리하게 활용할 수 있다.
### 풀스택 배포
- 애플리케이션을 구성하는 다양한 계층을 모두 다룰 수 있는 능력이 있는 개발자
- DevOps에서는 풀스택 배포는 클라우드환경에서 애플리케이션 서비스 실행 환경 구성에 대한 배포를 함께 진행하는 것을 의미
- 크게 두가지 배포 방식이 있다
	- 분리 배포: 클라우드 환경 프로비저닝을 애플리케이션 서비스 배포와 분리하여 배포
		- 클라우드 환경에 대한 오케스트레이션이 가능하지 않다면 부리 배포해야 한다.
	- 풀 스택 배포: 배포 자동화 도구를 활용하여 애플리케이션 서비스와 클라우드 환경을 함께 배포하는 방식
		- 서비스와 환경에 대한 오케스트레이션이 자동화 도구에 의해 이루어 지고 배포된다.

- 풀스택 배포의 장점
	- 클라우드 환경에 개발, 배포할 때 매우 유용
	- 신속하면서 일관된 서비스 배포를 진행할 수 있다.
	- 단위 서비스 개선이나 콘테느 추가 등에는 풀 스택 기반의 클라우드 배포를 쉽고 바르게 할 수 있다
### DevOps를 위한 클라우드 서비스 모델
- 클라우드 유형
	- IaaS (Infrastructure as a Service)
		- 가상화된 인프라만 제공
		- 운연체제, 미들웨어, 애플리케이션 설치 배포, 패치 드으이 관리 및 책임은 사용자에게 있다
		- 개발과 인프라 관리자간  사일로가 존재하여 DevOps의 이점을 얻기 어렵다.
	- PaaS (Platform as a Service)
		- 애플리케이션과 데이터의 생성 및 관리에 대해서만 고민하면 된다.
		- Paas모델에서는 개발 지우너 도구, 테스트 도구, 배포를 위한 지원도 제공
		- 개발이라는 핵심 임무에 더 집중 할 수 있다.
	- SaaS (Software as a Service)

## DevOps 측정 정도
- 가능한 신속하고 지속적으로 코드를 배포하는 것이 목적이다
	- 프로세스 상 오류나 잘못된 점들이 발생하지 않아야 한다.
- 매트 왓슨이 제한 다음 척도를 통해 DevOps의 성과를 분석, 추척할 수 있다

- 설치 주기
	- 사용자 환경 설치 주기를 추적
	- 가능한 더 작은 규모릐 배포를 수행
	- 배포 크기를 줄이면 테스트 및 릴리즈가 쉬워진다.
	- 작은 규모는 결함을 찾기 쉽고 결함 관련 척도를 낮추기 좋다.
-  설치 규모
	- 사용자 요구사항의 양, 요청된 기능의 규모, 수정된 버그의 양 등을 추적
	- 개별 작업 항모의 크기에 따라 설치 규모가 크게 달라 질 수 있다.
	- 며칠 분량의 개발 작업이 배포되고 있는지 추적할수 있다.
- 설치 시간
	- 실제 배포 및 설치를 수행하는데 걸리는 시간
	- 배포 설치하는 걸리는 시간 = 서비스 중단 시간
	- 설치 시간이 줄어들면 더 자주 배포하고 설치하는데 유리하다.
- 소요 시간
	- 작업을 시작하여 배포될때까지 소요되는 시간을 정의
	- 작업 항목이 시작되고 완료되는 걸리는 시간을 알 수 있음
- 고객 티켓
	- 고객티켓은 애플리케이션 서비스에 대한 고객이 제시하는 문제를 관리하기 위한 식별 수단
	- 품질 및 성능과 관련된 지표를 제공
- 자동 테스트 성공률
	- 단위 및 기능 테스트를 광범위하게 수행하여 결함을 자건에 제거하는 것이 좋음
	- 테스트의 성공률을 추적하면 테스트에 대한 소요시간이나 시간 지연을 예측할 수 있다.
- 결함 제거율
	- 문제 유발 결함이 완제품에 남게되는 빈도를 추적
- 가용성
	- 개발이 완료된 뒤 배포, 설치 과정에서 배포 방법이에 따른 제야기나 설치 과정의 문제로 인하여 시스템에서 일시적인 중단이 일어날 수 있다.
	- 끊이 없는 서비스를 제공하기 위하여 시스템 가용성을 추적
- 서비스 수준 협약
	- 서비스 제공자가 서비스 가입자에게 사전에 정의된 수준의 서비스를 제공하기로 상호 체결한 협약이다.
- 서비스 활용 및 트랙픽
	- 배포 설치 후 시스넴에 접근하는 트랜젝션 또는 사용자 수가 정상인지 아닌지를 확인해야 한다.
	- 갑자기 트래픽이 없거나 급증하면 문제가 있다고 볼 수 있다.
- 서비스 성능
	- 배포를 수행하기 전에 자동화 도구를 이용ㅇ하여 애플리케이션의 성능 문제, 숨겨진 오류 및 기타 문제를 찾아야 한다.
	- 배포 중이거나 설치 중에도 전체 서비스의 성능 변경 요인을 확인해야 한다.
	- 배포 및 설치 후 에도 특정 쿼리, 웹 서비스 호출등으로 인하여 서비스 성능에 변화가 발생하는지 확인해야 한다.
- 결함 발견 간 평균 시간
	- 문제가 발생하면 신속하게 식별하는 것이 중요
		- 최악의 경우 시스템 중 단 혹은 전체 시스템 중단으로 이어짐
	- 모니터링 도구를 활용하여 결함으로 인한 시스템의 오작동을 신속히 발견하고 수정 대처해야 한다.
- 결함 복구 간 평균 시간
	- 장애를 복구한데 걸리는 시간을추적하는 일은 서비스의 가용성을 높이고 사용자불만을 줄이는 데 중요
- 장애원인을 신속하게 파악하고 복구하여 시간을 줄여야 한다.
## DevOps 변형
- 개발 프로세스에서 DevOps의 성과가 입증되고 더 나아가 빅데이터 처리 관리, 보오나 등의 영역에서도 DevOps의 효과를 발휘할 수 있는지에 대한 궁금증
### DevSecOps
- DevOps의 접근 방식에 보안 사례를 통합할 수 있도록 DevOps를 확장한 것
- 소프트웨어 생명 주기의 ㅁ도느 단계에서 자동으로 보안을 적용하여 보안 소프트웨어를 개발할 수 있도록 한다.
- 기존의 방식
	- 별도의 보안 팀에 의해 개발주기의 마지막 단계에서 반영
	- 1년에 한 두번 진행되었을 때 보안 이슈를 관리
- 보안에 대한 기존의 접근 방식으로는 애자일과 DevOps를 수용할 수 없고 병목 현상이 발생

- DevSecOps
	- DevOps 프로세스 및 도구와 통합함으로써 원활한 보안을 지원
	- 보안을 단일 사일로 단독 책임이 아닌 개발, 보안, 운영팀의 공동의 책임으로 정의함
	- 개발 주기를 지연시키지 않으면서 자동화 환경에서 보안 소프트웨어를 제고
	- 보안 취약점 패티를 신속하게 수행 할 수 있다.
#### ShiftLeft
- DevSecOps의 핵심
- 왼쪽으로 이동하면서 보안관련 이슈를 처리하도록 권장
- 사이버 보안 걸계자와 엔지니어의 임무
	- 모든 구성 요소와 스택의 구성 항목이 보안 위협으로부터 안정하게 패치 되고 결과를 문서화 하는 것
- DevOps = 왼쪽에서 오른쪽으로 이동하면서 제품을 효율적으로 구축하고 보안 기능도 구현
- DevSecOps는 오른쪽에서 왼쪽으로 이동하며 보안 위험과 노츨을 조기에 식별하고 즉시 해결

#### 보안 교육
- 조직 내 모든 사람들은 조직의 봉나 상태를 이해하고 동일한 보안 표준을 준수해야 한다.
- 보안의 기본 원칙, OWASP의 상위 10개의 보안 리스크, 애플리케이션 보안 테스트 및 기타 보안 엔지니어링 관행에 익숙해야 한다.
- 개발자는 보안 규정 준수 검사 방법을 이해하고 보안 위협 노출 여부를 특정하겨 봉나 제어 구현 방법을 제시해야 한다.
- 보안에 대한 전반적인 실무 지식을 학습하는 것이 중요하다.

- OWASP에 대한 사위 10가지 보안 리스크 목록
	- Injection
	- Broken Authentication
	- Sensitive Data Exposure
	- XML External Entitied
	- Broken Access Control
	- Security Misconfiguration
	- Cross-Site Scripting
	- Insecure Deserializatino
	- Using Components with known Vulnerabilities
	- Insufficient Logging & Monitoring
#### 상호 소통문화
- 개발 프로세스와 제품 소유권에 대한 보안 이슈의 책임을 정확하게 알려주는 것이 매우 중요하다.
	- 그래야만 개발자와 보안 엔지니어가 개발 프로세스의 소유자가 되어 작업에 대한 책임을 질 수 있다.
- 현재 프로젝트에 적합한 기술과 프로토콜을 사용하여 자신들에게 적합한 보안 체계를 구축해야 한다.
- 팀에서 필요한 워크플로우 환경을 만들 수 있도록 허용함으로써 프로젝트 관리 결과에 대한 관심을 갖게 된다.

#### 추적 가능성, 감사 가능성, 가시성
- 추적 가능성
	- 개발 주기 전반에 걸쳐 요구사항이 구현되는 형상 항목을 추적
	- 관리 규정 준수, 버그 감소, 안전한 코드 버전 관리, 코드 유지 관리에 도움
- 감사 가능성
	- 보안 제어 준수를 보장하는데 주용한 다
	- 기술, 절차 및 관리에 대한 보안 통제는 감사가 가능하도록 문서화 한다.
	- 모든 팀 구성원이 이를 준수해야 한다.
- 가시성
	- 조직에서 수행되는 작업 상태를 확읺측정하고 필요시 알람을 발생시킨다.
	- 프로젝트 전체 수명 주기 동안 작업 역할과 책임을 제공하는 모니터링 시스템 역할 수행한다.
### DevDataOps
- DevOps 실현을 위한 수단인 자동화 및 툴체인은 다양한개발산출물과 함께 메시지, 로그, 데이터 등을 지속적으로 생성한다
	- 따라서 DevOps의 목표를 달성하기 위해서 데이터를 적절하게 관리, 수집, 처리, 저장하여 운영에서 개발 단계에 이르는 통찰력을 발위 할 수 있어야 한다.
- DevDataOps는 데이터 분석 수명주기의 품질과 정보제공의 적시성을 향상하는 것을 목표로 하는 새로운 접근법이다.
- 애자일 규칙 및 린 개념을 채택하여 데이터 분석에 DevOps의 이점을 제공한다.
- 데이터의 볼륨이 점점 더 커질때 팀간의 의사소통, 협력, 자동화, 통합등의 DevOps 원칙을 활용하여 데이터 분석의 수명 주기를 개선하는 것
- DevOps 프로세스에서 생성된 데이터는 매우 복잡하고 이질적이어서 데이터 유지 관리가 어렵다
	- 따라서 데이터 흐름으 ㅣ파이프라인에서 각 단계의 활동을 이해하고 데이터 분석활동에 통합하여야 한다.

데이터 흐름 파이프라인
- 생성
	- DevOps 프로세스의 각 활동은 특정 모니터링 과정과 도구를 통해 작업 상화 및 결과에 대한 로그 스트림을 생성해야 한다.
- 수집
	- 생성된 로그를 수집하고 타임 스탬프에 따라 정렬되고 그룹화되어 각 시간 간격에 대한 전체 DevOps프로세스의 스냅샷을 제공
- 필터링
	- 오류, 분석에 바람직하지 않은 이상 값, 중복 데이터 등을 제거하기 위해서 로그를 필터링한다.
- 스트림 처리
	- 단일 로그 스트림은 특정 DevOps프로셋 단계에서 잠재적인 결함 또는 오류를 식별하기 위해 실시간 로컬에서 처리된다
		- 이때 도구에 의한 경고, 오류 메시지 , 요구되는 기타활동이 촉발된다.
- 집적
	- 통계 분석을 위해 로그를 취합하여 요약 형식으로 표현
	- 데이터 볼륨을 압축하는 것
- 배달
	- 최종 사용자와 응용프로그램의 로그를 네트워크를 통해서 물리적 서버및 가상 서버로 전송
- 정렬 및 그룹화
	- 타임 스탬프에 따라 재정렬되고 그룹화되어 시간 간격으로 스냅샷 정보를 제공
	- 중복을 제거하여 저장하기 전 사전에 정의된 방법에 따라 다른 소스와 스키마 통합이 이루어져야 한다.
- 저장, 갱신/보관
	- 효율적인 저장 및 구성과 사용 가능한 로그의 지속적인 업데이트를 목표
- 처리 및 분석
	- 저장과 보관된 로그에 대하여 단계 간의 상관 관계를 식별
	- 데이터의 의미있는 패턴에 대한 발견, 해석 제공을 지원
		- 무엇이 발생하고 어떠한 데이터가 발생 가능성이 있는지 어떤 조치를 취해야 하는지 분석한다.
#### 구현을 위한 고려 사항
- DevOps상 발생하는 데이터를 처리하기 위하여 데이터와 과련된 도구를 식별하고 워크플로우와 연결한다
- 빅데이터를 지우너하는 솔루션을 채택하여 효과를 얻을 수 있다.
- 데이터 활용및 관리릉 위한 병렬화와 최적화를 고려하여 진행한다
- 클라우드와 같은 활경에 배포할 수있어야 한다.
- 운영 단계의 모니터링 대상에 대한 척도가 정의되고 모니터링을 통한 데이터 수집 도구가 지원되어야 한다.
- 운영단계의 데이터가 충분히 피드백 될 수 있는 경로를 제공해야 한다.
- 데이터 엔지니어가 피드백되는 데이터를 분석할 수있는 도구가 필요하다.
### BizDevOps
- DevOps 2.0
	- 개발자, 운영 직원, 비즈니스 팀의 협업이 중요
- 팀간 협업을 통해 조직이 소프트웨어를 더 빠르게 개발하고 사용자요구에 더 잘 대응할 수 있게 하며 궁극적으로 수익을 극대화 할 수 있도록 유도하는 개발 접근 방식
- 기존의 개발, 운영, 비즈니스의 사일로 벽을 허물고 소프트웨어 개발 방식을 근본적으로 변경
- 비즈니스 팀은 요구사항을 설정하고 개발자와 직접 협력항 ㅕ개발 스프린트 및 백로그의 우선순위를 설정한다
	- 문제를 해결하면서 비즈니스 목표를 달성한다.
- 실시간 분석의 중요성
	- 성능 모니터링 도구와 분석 도구를 사용하여 애플리케이션 성능 및 최종 사용자 반응에 대한 데이터를 즉시 얻고 이 데이터가 비즈니스KPI를 얼마나 잘 달성한지 정량화 한다. 
	- 비즈니스 목표가 애플리케이션 개발 및 배포의 작은 세부 작업에서도 고려되고 작업 방향을 가이드 한다.
- BizDevOps 프로세스
	- 비즈니스 목표와 핵심 성과 지표를 공유한다
	- 비즈니스 문제 정의 -> 코드 개발 -> 코드 패키징 -> 테스트 환경으로 제공 -> 코드 테스트 -> 운영 환경으로 제공 -> 모니터링 및 통제
		- 피드백 루프를 통한 다시 비즈니스 문제를 정의
	- 각 단계마다 비즈니스 목표 및 핵심 성과 지표를 보고 및 공유
## DevOps와 기술 부채
- 기술 부채
	- 시간이 오래걸리는 더 나은 접근 방식보다 지금 당장 사용할 수 있는 쉬운 솔루션을 선택함으로써 발생하는 추가 재작업의 내재된 비용
	- 이자가 누적 되어 변경사항을 구현하기 어려워짐
		- 코드의 변경이 발생하는 경우 코드나 문서의 다른 부분에서 변경이 발생한다.
		- 이때 완료되지 않은 변경사항을 부채로 간주
		- 부채를 털어버릴때 까지 이자에 이자가 붙는 방식
### 기술 부채의 원인
- 지속적인 개발
	- 지속적인 변경과 최적의 솔루션을 만들어 내기 위한 과정의 반복
- 불충분한 정의
	- 요구사항이 충분히 정의되지 않은 상황에서 시간을 절약하기 위해 설계가 진행되는 경우
	- 종종 나중에 다시 작업해야하는 경우가 발생
- 비즈니스 압력
	- 변경이 완료되기 전에 비즈니스 목적상 릴리즈를 빨리 진행하는 경우 완료되지 않는 변경과 이와 과련된 기술적 부채가 쌓인다
- 프로세스 이해 부족
	- 개발 조직에서 기술 부채의 개념을 이해하지 못하고, 의사결정을 내리는 경우
- 결합력이 높은 구성 요소
	- 기능이 모듈화되지 않은 경우 요구사항의 변화를 반영할 만큼 충분히 유연하지 못함
- 테스트 부족
	- 결함 혹은 버그를 수정하기 위하여 빠르지만 위험한 반창고식 수정을 진행
- 소프트웨어 문서 부족
	- 지원 문서없이 코드가 생성되는 경우 문서를 만드는 작업은 부채
- 협업 부족
	- 조직에 지식이 공유되지 않아 프로젝트 수행의 효율성이 떨어진다
	- 주니어 개발자가 제대로 멘토링을 받지 못한 경우 문제 발생
- 병행 개발
	- 변렬 개발 수행중 변경이 발생하느 경우 소스 통합 과정에서 더 많은 오력을요구
	- 개별적이 변경이 많을 수록 부채는 지수적으로 증가
- 지연된 리팩토링
	- 리팩토링이 지연되고 코드에 추가되면 부채가 커진다
- 표준에 대한 조정 부족
	- 개발 과정에서 산업 표준이 무시되는 경우 통합 과정에서 표준 적용을 위한 노력 증가
- 지식 부족
	- 융통성 있는 아키텍처 설계 혹은 아우한 코드 작성법을 모르면 유지보수 비용이 증가한다.
- 소요권 부족
	- 코드를 리팩토링할 려고 할 때 다시 작성해야 하는 경우
- 막판 사양 변경
	- 종료 시점에 변경이 발생하는 경우, 이를 반영하기 위한 오력과 시간을 출분히 등리기 어렵고 테스트 또한 충분히 이루어 지지 않는다.
### DevOps에서 발생하는 기술 부채
- DevOps를 도입하기 시작하면서 전반적인 기술 부채를 개선해야할 필요성이 커짐
	- 예시
		- 작업이 연기되는 경우 관심과 노력이 줄어들면서 일을 빨리 처리하려고 하면서 기술의 부채가 발생한다
- DevOps는 개발 프로세슷를 자동화하여 발생 가능한 기술 부체에 대응하고자 한다.
	- 수작업에서 발생할 수 있는 문제를 제거할 수 있다.
- DevOps 파이프라인과 상호작용하지 않거나, 배포,운영에 영항을 미치지 않느 기술 부채 개발자가 해결해얗할 문제로 남게된다.
	- 이러한 경우 기존의 개발 관행을 완전히 제거하지 못하면 이 또한 기술 부채로 남게된다.
- 기술 부채를 완전히 제거하는 방법은 없다
	- 어느정도 수용하고 감내하면서 최적의 해결 방안을 모색해야 한다,.
### 기술 부채 절감을 위한 DevOps 전략
- DevOps에도 기술 부채가 존재하기 때문에 '어떻게 기술 부채를 통제하고 줄일 것인가'는 중요한 문제이다

- 동기 부여
	- 기술 부채의 대상이 되는 요소를 식별하고, 해당 요소의 수준을 평가하는 것
		- 이를 통해 해결해야할 기술 부채를 개발을 위한 하나의 스프린트로 배정
	- 관리자는 식별된 기술 부채를 공유함으로써, 재발생을 줄이거나 새로운 기술 배채가 발생하지 않도록 관리한다.
- DevOps 자동화 활용
	- 자동화를 통해 미래의 부채를 피하고 현재의 부채를 상쇄해야 한다
	- 일상적이고, 오류가 발생하기 쉽고, 시간이 많이 소요되는 프로세스는 자동화 하면 기술 부채를 완화할 수 있다.
- 마이크로서비스 모델 채택
	- 애플리케이션의 현대화하는 가장 쉬운 방법은 비저장 마이크로서비스를 사용하는 것
	- 기존 프로세스 간의 종속성을 유발하지 않는 기때믄에 서비스 간에 공유 데이터 저장소 및 리소스의 공유가 필요하지 않는다
	- 단일 기능으로써 변경되더라도 다른 기능에 영향을 미치지 않는다.
	- 따라서 신속한 변경과 높은 릴리즈 빈도가 허용되므로 변경에 대한 기술부채에서 자유로워 진다.
- 컨테이너화된 배포
	- 컨테이너화를 통해 지속적인 배포를 패키징한다면 수동 개입없이 변경된 증분 코드를 배포할 수 있다.
	- 컨테이너 아키텍처에 대한 패키징 서비스가 개발자의 할 일을 덜어준다
- API 중심 모델 구축
