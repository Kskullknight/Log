# 실현 가능성 분석
- 소프트웨어 개발을 위한 첫번째 단계이다.
## 실현 가능한가?
### 비용(Cost)
- 개발에 드는 인건비
	- 공수 기반: 투입 인워수 X 투입 기간(MM: Man Month) X 단가
		- 단가는 직급, 등급별로 다름
		- MM: 한달(20~22일)동안 전담하여 수행할 수 있는 일
	- 기능 점수 기반
		- FP(Function Point): 소프트웨어 개발 규모를 측정하기 위해 기능을 정량화하고 계수적 측정을 통해 나타낸 수치 또는 기법
> [!faq]+ 기능 점수 기반 측정 방법
> 1. 가용문서 수집
> 	- 기능점수 측정을 위해 SW 제공 기능, 측정 대상 PJT에 의해 영향 받는 기능을 설명할 수 있는 문서 수집
> 2. 측정 범위및 경계 설정
> 	- 어플리케이션의 측정범위와 경계 결정
> 	- 사용자 기능 요구사항 식별
> 3. 기능 측정
> 	- 데이터 기능 측정
> 	- 데이터 저장, 참조하는 사용자 요구사항 충족 기능 측정
> 	- 구성 요소
> 		- 내부 논리 파일 (ILF: Internal Logical File)
> 			- 측정범위 내 유지되는 논리적 데이터 그룹/제어 정보
> 			- 간이법 평균복잡도: 7.5
> 		- 외부 연계 파일 (ELF: External Interface File)
> 			- 측정범위 밖 타 APP 참조 논리적 데이터 그룹
> 			- 간이법 평균복잡도: 5.4
> 	- 트랜잭션 기능 측정
> 		- 데이터 처리의 사용자 요구사항 기능 측정
> 		- 구성 요소
> 			- 외부 입력 (EI: External Input)
> 				- APP 외부에서 데이터/제어 정보 수신하여 상태 변경 프로세스
> 				- 간이법 평균복잡도: 4.0
> 			- 외부 출력(EO: External Output)
> 				- APP 내부에서 경계 밖으로 내보내는 프로세스
> 				- 간이법 평균복잡도: 5.2
> 			- 외부 조회(EQ: External Query)
> 				- APP 내부에서 데이터/제어 정보를 외부로 내보내는 프로세스
> 				- 간이법 평균복잡도: 3.9
> 	- 기능 점수 측정 방법
> 		- 간이법
> 			- 기능 유형별 복잡도 적용하여 산출
> 			- 평균 복잡도 적용
> 			- SW 기획 및 발주 단계
> 			- 측정 간소화, 신속한 규모 측정
> 			- 유사사업 재활용 제한, 낮은 정확도
> 		- 정통법
> 			- 기능 도출 후 복잡도 고려하여 정확한 기능점수 산정
> 			- 기능별 복잡도 계산하여 적용
> 			- SW 설계 공정 이후
> 			- 정확한 규모 측정, 유사 사업 재활용
> 			- 사업초기 적용 어려
> 4. 기능 규모 계산
> 	- 측정의 목적과 범위 고려 계산
> 	- 개발 및 개선 프로젝트의 기능 규모 계산
> 5. 기능 점수 문서화 및 보고
> 	- 측정 목적/유형/범위/경계 포함 문서
> 	- 데이터/트랜잭션의 기능 리스트, 복잡도 측정 결과 보고

- 하드웨어 비용
	- 자체구축
	- IaaS, PaaS
- 개발 및 운영에 필요한 소프트웨어 도구 비용
	- 라이선스 구독비용
- 유지 보수 비용
	- 초기 개발비의 10%~20%
	- 연 유지보수율 X 개발비 X 운영 연수
- 문서화 소요 비용
	- 인건비 공수에 포함
	- 5% ~ 10% 배정
- 개발자 교육 비용
	- 신기술 도입시 학습 기간 동안의 인건비 저하도 비용으로 잡을 수 있다
- 시스템 교체 비용
	- 데이터 마이그레이션 비용
	- 신규 시스템 구축비
	- 병행 운영기간의 이중비용
	- 기존 시스템 페기 처분 비용
- 등등 소프트웨어 개발에 필요한 모든 비용
	- 외주, 컨설티 비용
	- 회의, 관리(PM) 간접 비용
	- 리스크 대응 예비비
### 이득 (Benefits)
- 새로운 시스템이 제공하는 비즈니스 프로세스의 개선된 영역은 무엇인가?
- 운영의 효율성과 정확성의 증진 정도
- 비용 절감 효과
### 대안(Alternatives)
- 비용과 이득 산정 시 다양한 측면을 고려
- 하나의 문제에 대한 여러 대안을 고민
## 어떤 측면을 고려할 것인가
- 실현 가능성을 분석하기 위한 다양한 측면

- 경제적 측면
	- 비용 대비 효과
- 기술적 측면
	- 이론이나 기술적 제약
- 스케줄 측면
	- 주어진 기간 내에 완료할 수 있는가?
- 운영적 측면
	- 운영에 대한 역활이나 책임
- 동기적 측면
	- 사용자에게 필요성이 있는가?
	- 정확하게 업무에 적용할 수 있는가?
- 법, 윤리적 측면
	- 사회적인 문제를 유발할 수 있는 기능이 포함되어 있는가?

# 전통적인 소프트웨어 프로세스
- 정의
	- 소프트웨어 개발 과정을 체계적으로 계회하고 관리할 의도 아래 서로 관련있는 활동을 단계로 분리하여 구분한 것
	- 무엇을 수행해야 하는가?를 정의한 일련의 단계
- 복잡하고 규모가 큰 개발일 수록 더울 준수해야하는 과정
## 소프트웨어 프로세스의 필요성
- 블랙박스 프로세스
	- 하나의 통합된 개발 과정을 거처 소프트웨어를 개발하는 경우
	- 사용자의 요구사항을 만족시키지 못할 확률이 높아져 과도한 수정이나 재개발을 진행할 수 있다.

- 투명한 프로세스 (Transparent Process)
	- 개발 과정을 여러 단계로 분리
	- 각 단계의 종료 시점에서 사용자에게 요구사항을 대한 확인 가능
		- 사용자의 요구사항을 만족시킬 확률을 증가 시킴
	- 중간 결과물 확인 가능
		- 재품 품질을 중간 중간 확인 가능
		- 어느정도 품질을 보장
	- 수정, 재개발 비용 절감
	- 스케줄, 리스크 관리
## 전통적인 프로세스 모델
### 폭포수 모델 Waterfall Model
- 물이 떨어지는 것처럼 순차적인 단계로 개발

- 각 단계마다 정해진 소프트웨어 문서를 작성
	- 문서는 사용자 확인 과정을 거처 다음 단계의 입력으로 사용

- 단점
	- 개발 과정에서 요구되는 변경을 수용하기 어렵다
	- 오류의 전파

### 점진적 모델
- 피드백 가능한 폭포수 모델
- 개발 범위를 관리 가능한 일정 크리의 그룹으로 분할
	- 개발 및 배포: 실사용자가 원하는 것을 개발하여 제공
	- 측정 및 모니터링: 배포된 부분에 대한 사용자의 유용성 평가
	- 조정 및 수정: 모니터링 결과를 실제 설계와 구현에 반영

- 장점
	- 변경을 쉽게 수용할 수 있다
	- 사용자에게 신속하게 사용 가능한 소프트웨어
- 단점
	- 반복적인 테스트와 통합 수행이 오버헤드로 작용할 수 있다
	- 사용자 피드백을 반영하다가 기존의 시스템 구조가 무너질 수 있다.
	- 일부 기능 배포가 사용자에게 실망을 가져다 줄 수 있음

#### 프로토 타입 모델 (Prototype Model)
- 사용자 인터페이스나 핵심 모듈만 우선적으로 개발하여 사용자에게 제공
	- 프로토타입 제작
- 프로토타입을 통해 사용자의 요구사항을 만족했는지 여부를 판단

- 장점
	- 사용자 요구사항을 검증하여 개발 비용과 시간을 단축
	- 사용자와 개발자 간 동일한 개념 확보
	- 빠른 오류 탐지
- 단점
	- 사용자 검증 과정이 후에 변경이 발생하는 경우 변경을 적용하기 힘들다

#### 나선형 모델
- '목표 설정 -> 위험 분석 -> 구현 및 테스트 -> 평가 및 단계 수립 활동'의 반복
- 시스템 개발시 위험 최소화
- 단계
	- 목표 설정
		- 고객의 요구사항 분석 및 타당성 검토
		- 각 단계의 활동에 해당하는 목표를 설정
	- 위험 분석
		- 고객 요구사항을 기반으로 예측되는 위험사항을 추출
		- 위험 사항에 대한 대처 방안 수립
		- 위험을 조기에 발견하여 영향을 최소화
	- 개발 및 검증
		- 소프트웨어 구현
		- 테스트 수행
	- 평가 및 다음 계획 수립
		- 산출물을 평가하고 다음 활동의 입력으로 사용

- 장점
	- 체계적인 위험 관리
	- 상세한 사용자 요구사항을 적용
	- 변경되는 요구사항을 반영하기 쉽다
	- 사용자 만족도와 품질 상승
- 단점
	- 프로젝트 기간이 길다
	- 반복되는 사이클이 많아 질 수록 프로젝트 관리가 어렵다
	- 위험 분석에 대한 비용과 전문 지식 요구

#### V 모델 (V Model)
- Verification and Validation 모델
- 개발 생명 주기와 단계별 테스트 활동 간의 관계를 정의
- 개발 활동과 테스트 활동 간의 대응 관계를 각 개발 단계에서 문서화
	- 개발 활동이 종료되는 시점 테스트 계획을 결정

- 장점
	- 단계적인 검증과 확인 과정을 통해 오류를 줄일 수 있다
	- 요구사항이 정확하고 작은 시스템 개발시 유용하다
- 단점
	- 반복이 허용되지 않기 때문에 변경을 다루기 쉽지 않다.

#### 변환 모델 (Transformation Model)
- 정형 명세를 기반으로 하는 소프트웨어
- 소프트웨어 요구 사항을 정형 기법을 이용하여 명세한 후 자동화 도구를 이용하여 코드로 변환
- 전문적인 정형 기법에 대한 전문적인 지식 요구
- 요구사항 단계 부터 수하적 개념을 바탕으로 이루어지기 때문에 안전 중심 소프트웨어나 임무 중심 소프트웨어를 개발하는데 사용

>[!note] 정형 기법이란?
>소프트웨어나 하드웨어 시스템을 수학적, 논리적으로 명세하고 검증하는 기법
>- 자연어 대신 수학에 기반한 엄밀한 표기법 사용
>
>핵심 아이디어
>- 모호성 제거: 요구사항을 수학적 기호로 적어 다른 해석의 여지를 제거
>- 완전한 검증: 수학적으로 입출력을 정의
>
>장점
>- 오류가 존재하지 않음을 보장
>
>단점
>- 높은 비용: 명세와 증명에 많은 시간을 소비
>- 전문성 요구

### 신속한 소프트웨어 개발
- 잦은  요구사항 변경에서 안정적인 요구사항 확보
	- 사용자의 참여를 통한 산출물 검토
- 명세 및 설계 단계에서 사용자 이해의 어려움
	- 사용자 인터페이스와 프로토타입을 통한 사용자와 의사소통
> 가시적인 소프트웨어를 통해 반복적인 사용자 검토 진행
#### 애자일 프로세스의 12가지 기본 원리
1. 고객 만족을 가능한 빨리 이끌러내고, 유현한 소프트웨어를 지속적으로 사용자에게 제공한다.
2. 개발 과정에서 늦ㅇ느 시점일지라도 요구사항 변경을 적극 수용한다.
3. 실행 가능한 소프트웨어를 1~2주에 한번씩 사용자에게 배포한다.
4. 사용자와 개발자 간 친밀한 미팅을 겅의 매일 수행한다.
5. 프로젝트는 신뢰할 수 있고 동기가 부쳐된 개인을 주심을 ㅗ진행한다.
6. 의사소통을 위해 가능하면 한 곳에 모여 대화한다.
7. 실행가능한 소프트웨어를 기준으로 프로젝트 진척률을 결정한다.
8. 일덩한 개발 소도를 유지하면서, 지속가능하도록 소프트웨어를 개발한다.
9. 우수한 기술의 도입과 좋은 설계에 지속적인 관심을 갖는다.
10. 단순화는 필수다. 작업을 가장 단순화 형태로 분리하여 괴민아고 해결한다
11. 최상의 아키택처 정제된 요구사항, 좋은 설계는 잘 구성된 팀을 통해 개발한다.
12. 팀은 더 효과적인 방법의 적용 적절한 조치 과정을 정기적을으로 수행한다.

### 애자일: XP 프로세스
- 반복 개발 접근 방법
	- 새로운 버전의 소프트웨어 개발
	- 2주에 한번 사용자에서 배포
	- 철저한 테스트를 통과한 빌드만 배포가 허용
- 사용자에게 더욱 신속하고 정확하게 소프트웨러르 제공하는 것을 목표

#### 개발 사이클
- 엔지니어링 사이클
	- 개발 작업에 대한 사이클
	- 코딩 -> 테스트 -> 사용자 피드백 -> 설계 -> 코딩

- 배포 사이클
	- 사용자 스토리 선정 -> 스토리 분할 및 태스크 식별 -> 배포 계획 수립 -> 개발/통합/테스트 -> 소프트 웨어 배포 -> 시스템 평가 및 피드백

#### 애자일: 스크럼 프로세스
- 반복적인 개발 관리에 초점
- 사용자에게 배포 가능한 소트프웨어를 반복적으로 개발하고 배포
	- 사용자을 요구사항을 지속적으로 반영

>[!note] XP와 스크럼의 차이
>- XP: 소프트웨어 개발의 기술적 실천(엔지니어링 프랙티스)에 집중
>- 스크럼: 프로젝트 관리 프레임워크로, 팀 운영과 진행 관리에 집중

##### 스크럼 프로세스 용어
- 스프린트: 반복적인 개발 주기
- 제품 백로그: 제품에 대한 요구 사항 목록
- 스프린트 백로그: 스프린트 목표에도달하기 위해 필요한 작업 목록
- 제품 증분: 스트린트 결과로 나오는 실행 가능한 제품
- 번다운 차트: 스프린트 백로그에 남아있는 작업 목록
- 번업 차트: 배포에 필요한 진행 사항을 보여주는 차트
- 제품 책임자: 백로그의 정의하여 우선순위를 정하는 책임자
- 스크럼 마스트: 프로젝트 관리자

##### 스크럼 프레임워크
- 솔수션에 포함할 기능 및 개선점에 대한 우선순위를 부여
- 개발주기를 30일 정도로 하며, 개발 주기마다 실제 동작하는 결과를 제공한다.
- 개발 주기마다 적용할 기능이나 개선에 대한 목록을 제공한다.
- 날마다 15분 정도 회의하라
- 항상 팀 단위로 생각하라
- 원활한 의사소통을 위해 구분없는 열린 공간을 유지하라

##### 스크럼 프로세스
1. 관리자, 팀, 스테이크 홀더, 고객으로부터 입력
2. 제품 관리자 => 제품 백로그: 요구사항, 스토리, 우선순위 목록
3. 개발팀 => 스프린트 계획: 제품 백로그에서 한 스프린트에서 개발 완료할 수 있는 작업 선택
4. 스프린트 백로그: 분류된 백로그
5. 스프린트
	- 스크럼 마스터
	- 번업다운 차트
	- 매일: 스크럼 미팅
6. 종료된 작업=> 스프린트 리뷰, 회고

>[!note] 스크럼 미팅을 진행하는 이유
>1. 진행 상황 공유: 어제 한 일, 오늘 할 일, 장애물
>2. 장애물 조기 발견: 문제가 커지기 전 미리 발견
>3. 스프린트 목표를 향한 매일의 점검과 조정: 검토와 적응의 반복
>4. 동기화 및 책임감: 팀의 리듬과 맡은 일에 대한 책임

## Chaos와 DevOps
- DevOps란
	- 개발과 운영 활동에 대한 기민한 협업과 활동의 자동화를 통한 진화된 소프트웨어 개발 활동을 의미
### Chaos Engineering
- 분산 시스템에서 의도적으로 장애를 주입해, 시스템이 예상치 못한 상황을 견디는 능력을 검증하는 실험적 방법론
	- 통제된 환경에서 장애를 발생시켜 미리 경험하자
#### 실험 절차
1. 정상 상태 정의
	- 시스템이 정상일 때 츨정 가능한 지표
		- 초당 처리율, 에러율, 응답 시간
2. 가설 수립
	- A장애가 발생해도 정상 상태가 유지될 것이다
3. 장애 주입
	- 현실에서 일어날 법한 변수를 주입
		- 서버 종료, 네트워크 지연, CPU 과부화, 패킷 손실, 자원고갈, 전체 다운 시뮬레이션
4. 결과 관찰/검증
	- 정상 상태가 유지됐는지 측정
	- 깨졌다면 약점을 발견한 것
5. 개선
	- 발견된 취약점을 보안

#### 중요 원칙
- 영향 범위 최소화: 작은 범위에서 시작해 점진적으로 확대
- 가설 기반의 통제도니 실험: 과학적 실험
- 자동 중단 장치: 실험이 위험해지는 경우 중단 가능해야함
- 모니터링: 관측 가능성을 갖추어야함

#### 중요 도구
- Chaos Mokey
- Gremlin
- Litmus
- AWS Fault Injection Service
- Chaos Mesh
- 등등
#### 기대효과
- 단일 장애점
- 잘못된 의존성
- 부족한 장애 복구 로직
- 회복 탄력성

> [!note] 카오스 엔지니어링과 부화테스트의 차이
> 유사점
> - 시스템을 괴롭혀서 검증한다
>
>차이점
>- 카오스 엔지니어링 - "고장나도 버티나"
>	- 목적: 시스템의 회복 탄력성과 장애 대응 능력 검증
>	- 변수: 장애 주입 (서버 종료, 네트워크 지연, 의존 서비스 차단)
>	- 관심사: 장애 복구, 이중화, 단일 장애점, 자동 복구 로직
>
>- 부화 테스트 - "얼마나 버티나"
>	- 목적: 트래픽을 늘려가며 시스템의 성능과 용량 한계를 측정
>	- 변수: 부화를 키움 (동시 접속자 수, 초당 요청수, 데이터 량)
>	- 관심사: 처리량, 응답시간, 병목 지점, 한계점

## Dev Ops Practices
- 사일로 접근 방식에서 발생하는 문제를 해결하기 위하여 제안
- 문제
	- 개발 팀 간의 불충분한 협업
	- 의사소통의 미흡 혹은 미흡
	- 개발 일정의 지연
	- 운영상의 유사 문제에 대한 중복된 노력

### DevOps의 정의
- 목적: 개발과 운영의 단절을 없애고 지속적인 소프트웨어 운영이 가능 하도록 한다

- Wikipedia의 정의
	- 개발자와 운영자 간의 소통과 협업, 통합을 강조하는 개발 방법
	- 개발과 운영의 상호 의존성르 명확히함
	- 제품 혹은 서비스의 신고한 개발을 목표로 함
- IBM의 정의
	- 개발팀과 운영 팀간의 조정을 위하여 제시된 프로세스
- O'reilly의 정의
	- 시스템의 운영 인프라를 PaaS 형태로 프로그램의 일부가 됨
	- 운영을 개발 활동의 일부로 고려함
	- 운영자가 개발팀에 합류해야하는 소프트웨어 개발 방법
- IEEE의 정의
	- 개발과 관려된 스테이크홀더들이 시스템또는 서비스를 명세, 개발, 운영하고 시스템 전체 수명 주기에 걸처 지속적인 갭선을 도모하기 위한 협업 증긴 원리와 실천 사항들

### DevOps의 핵심 원리
- 운영환경에서 나타난 요구사항을 지속적으로 반영하고 이러한 과정에서 시스템의 중단 없는 가용성을 제공하기 위한 원리

- 자동화 (Automation)
	- 개발 환경에서 소프트웨어 동작이 배포 및 운영 환경에서도 동일하게 보장되도록
	- 개발 과정에서도 운영 환경을 고려
	- 운영 환경의 오류를 신속하고 일관성 있게 해결하기 위해
	- 자동화된 환경을 전체 수명 주기에 구축
- 반복 (Iteration)
	- 유지보수라는 개념보다
	- 운영 과정에서 모니터링된 요구사항을 시스템에 반영하여 반복 개발하는 형식으로 진행
- 자기 서비스 (Self-service)
	- 협업및 자동화 환경을 통해서 개발자오 운영자가 서로 방해하지 않고 독맂벅으로 일할 수 있도록 지원한다
		- 개발자가 운영환경을 자체 구축하여 신속한 개발 및 테스트를 수행할 수 있도록
- 지속적 개선 (Continuous Improvement)
	- 사용자의 피드백을 통해 소프트웨어 및 운영 환경을 개선
	- 사용자의 피드백은 기능 향상, 추가, 삭제, 서비스 성능등 개선의 근간이 된다
- 협업 (Collaboration)
	- 신속하고 신뢰성 있는 소프트웨어의 개발과 배포를 보장한다.
	- 개발팀과 운영팀 간의 지속적인 협업 필요
- 지속적 테스트(Continuous test)
	- 개발자 - 단위 테스트와 통합 테스트
	- 품질 보증팀 - 시스템 테스트 수행
	- 운영팀 - 인수 테스트, 안정성 테스트 수행
	- 지속적인 통합과 배포 과정에서 이러한 테스트간 반복적으로 발생한다.
- 지속적 통합과 지속적 배포(Continuous Integration and Continuous Delivery)
	- 각 팀의 결과물이 주기적, 매일 통합한다
	- 올바르게 통합된 컴포넌트는 테스트 -> 생산 -> 사용자에게 전달
	- 배포도 문제가 없는 한 지속적으로 이루어진다.

### DevOps 프로세스
- 개발과 운영을 연결하는 프로세스
- 요구사항 -> 개발 태스크로 분할 -> 개발 배포 계획 수립 -> 컴포넌트 개발 -> 통합 -> 빌드 -> 테스트 -> 릴리즈 -> 배포 -> 운영 -> 모니터링 -> 계획 수립
### DevOps의 이점
- 제품 출시 기간 단축
	- 운영 팀의 피드백을 즉시 합병 할 수 있도록 협업한다.
	- 코드 배포 의도 시점과 코드 생성 시간 축소
- 낮은 위험과 유연한 배포
	- 개발 환경을 고려한 자동화된 플랫폼상에서 지속적으로 문제의 원인을 분석하고 대처 가능하다
- 신속한 복구
	- 자동화된 진단 도구를 통하여 문제 복구
	- 자동화된 테스트와 릴리즈 환경을 통해 문제 복구를 신속하게 가능하다
	- 롤백, 형상 매개변수의 변화등을 빠르게 적용 가능
- 고객 만족도 및 시장 적합도 향상
	- 신속 정확한 배포통해 피드백 수집하고 적용 후 빠르게 배포 가능
		- 이를 통한 고객의 만족도 향상

## 소프트웨어 프로세스 개선
- 소프트웨어 개발과 관련된 모든 활동을 더울 체계적으로 수행함으로써 소프트웨어의 결함을 줄이고 더 품질 좋은 제품을 만들기 위함
	- 개발 과정의 개선할 점을 고민
	- 개발 능력과 상황을 정확히 진단
	- 이를 기반으로한 개선 활동
- 소프트웨어 프로세스를 개선하기 위한 가이드라인과 표준
### CMM과 CMMI
#### CMM (Capability Maturity Model)
- 미 국방부가 수집한 데이터를 바탕으로 체계화하여 개발한 개발 프로세스의 성숙 모델
- 성숙도: 개발 단계별 활동에서 얼마나 잘하는지 나타내는 척도
- 성숙도를 바탕으로 기존 개발 프로세스를 개선하는 것을 목표

- 문제점
	- 소프트웨어 개별 활동과 시스템 획득 업무 간 연계성 부족
	- CMM을 구성하는 다양한 모델의 차이점으로 인한 통합 적용의 어려움

> CMM의 단점을 해결한 CMMI개발

#### CMMI
- 프로세스 기반 활동 영역(무엇을 하는가)과 능력도 범위(얼마나 잘하는가)로 표현
- 1~5 수준이 존재하고 각 수준에서 필요한 프로세스 영역의 역량 수준이 일정 수준에 도달하면 해당 프로세스를 해당 수준으로 평가한다.
- 용어 정리
	- 프로세스 영역: 관련 실천 행동들의 집합
	- 성숙도 레벨: 프로세스 성숙 정도를 단계별로 표현
	- 역량 수준: 개별 프로세스 영역의 성숙 정도를 단계별로 표현
- 활동 영역에 따라 구분됨
	- 개발
	- 서비스
	- 공급 관리
	- 사람
- 모델별 평균 22개 수준의 프로세스 영역이 있으며 대체혹은 추가된다.
### CMMI 인증
- 각 프로세스 영역은 목표와 실천 사항들록 구성된다
- 정의된 실천사항을 수행하는가에 대한 증거를 기반으로 프로세스 영역의 성숙도를 확인한다.
- CMMI  2.0에서 프랙티스 영역으로 기본 구조가 변경됨
- CMMI 버전이 1.3에서 2.0으로 향상되면서 변환
	- 복잡성과 규모는 축소
	- 적용성관 운영성은 향상
		- 프랙티스 영역이 1,252개에서 404개로 축소
		- 프로세스 영역에서 프랙티스 영역으로 변경되면서 22개에서 20개로 축소
	- 각 영역에 대한 번주의 한계를 명확히 하고 영역 구현을 더 쉽게 가이드함

- Class A, Class B, Class C로 구분된
- Class B, Class C는 내부 심사 혹은 사전 진단의 성격
- Class A는 공식 심사원이 진행
	- 모든 실천 사항에 대한 활동의 증거를 수집하고 진위 여부를 판단
### SPICE && A-SPICE
- CMMI와 함께 소프트웨어 프로세스 개선을 위한 영역의 표준으로 정의
	- 프로세스의 성숙도를 정의
- ISO의 표준으로 제정
#### SPICE
- ISO 12207에서 정의한 프로세스 수명 주기를 기반으로 작성
- CMMI와 유사하지만 0단계부터 6개의 성숙도 수준을 정의
- 프로세스 속성을 사용하여 측정
	- 프로세스 성능
	- 성능과 관리
	- 산출물 관리
	- 프로세스 정의
	- 프로세스 배포
	- 프로ㅔ슷 측정
	- 프로세스 제어
	- 프로세스 혁신
	- 프로세스 최적화
- 각 프로세스 속성에 대한 일반 활동으로 구성됨
- 일반 활동의 수행 정도에 따라 4단계로 구분함
#### A-SPICE (Automotive SPICE)
- SPICE의 표준에 따라 자동차에 맞게 수정하여 재정됨
- 3개의 생명 주기를 가진다
- 각 생명주기는 8개의 포로세스 그룹으로 정의
- 각 프로세스 그룹에서 수행해야 활동을 제시한다.
- 5개의 등급으로 구분
### 식스 시그마
- 프로세스 개선을 위한 공학적 기술 및 지우너 도구를 제공하는 프로세스 관점의 접근 방법론
	- 결함의 원인을 식별하고 제거
	- 기존 제조 및 비즈니스 프로세스에 대한 변경을 최소화
	- 공학적 활동의 품질을 개선하는 것을 목표
- 경험적, 통계적, 측면의 품질 관리 방법을 사용
- 정규 분포의 표준 편차 내에서 결함 없는 결과의 비율에 해당하는 것
- 가치 항상을 추구
	1. 공정 주기 단축
	2. 결함 및 문제 감소
	3. 비용 감소
	4. 고객만족도 향상
	5. 이익 증가
	6. 등등
#### DMAIC와 DMADV 구성 단계
- 식스 시그마는 DMAIC와 DMADV를 기반으로 한다

##### DMAIC
- 시스템 정의
	- 고객의 목소리, 요구사항, 프로젝트의 목표를 설정
- 핵심 요소 측정
	- 현재 프로셋의 주요한 측면을 측정
	- 관련 데이터 수집
	- 현재 상태의 프로세스 능력을 측정
- 데이터 분석
	- 데이터를 분석하여 인과관계를 조사
	- 인과관계의 유형 파악
	- 모든 요소가 고려되었는지 점검
	- 관심 대상 경함의 근본 원인이 무엇인지 찾는다
- 프로세스 개선 및 최적화
	- 데이터 분석으 위한 실험 서례, 포카 요케, 실수 교정, 작업 표준화와 같은 기술을 사용하여 현재 프로세스를 개선하거나 최적화하여 새로운 프로세스 구축
	- 파일럿 실행을 토해 이과정을 조정
- 미래 프로세스 조정 관리
	- 편차가 추후 결함으로 연결되지 않도록 수정하는 과정을 반복
	- 통계적 방법에 의한 프로세스 제어, 진행 상태의 공지, 지속적인 모니터링등이 가능한 환경 구축
	- 원하는 품질 수준을 얻을 때까지 반복

> [!note] 포카 요케
> - 실수를 방지하도록 행동을 제한하거나 정확한 동작을 수행하도록 강제하는 제약 조건을 만들어 실패를 방지하는 방법

##### DMADV
- 기본적인 접근 방법은 DMAIC와 동일

- 설계 목표 정의
	- 고객의 요구 및 기업 전략과 일관성 있는 목표를 정의
- CTQ(Critical to Quality) 식별 및 측정
	- 제품 성능 프로세스 능력, 리스크 등을 식별하고 측정한다.
>[!note] CTQ란
>고객 만족을 결정짓는 제품이나 서비스의 가장 중요한 품질 속성을 의미한다. 고객의 모호한 요구사항을 측정 가능하고 관리 가능한 구체적인 수치로 변환한 지표
- 대안 분석
	- 품질 향상을 위해 가능한 대안들을 분석한다
- 선정 대안 설계
	- 가장 좋은 대안을 선택하고 적용하기 위해서 설계한다.
- 설계 검증
	- 설계를 검증하고 파일럿 프로젝트를 통해서 개발 프로세스를 구현하고 실행한다.

#### 식스 시그마 인증
- 조직 내 엔지니어에게 부여하는 프레세스 품질의 기술 수준 및 역량을 진단하는데 목적이 있음

- 단계
	- 그린 벨트
		- 식스 시그마의 필수 기법을 이해하고 이를 실천할 수 있는 인재에게 부여하는 인증 자격
		- 프로젝트 맴버 혹은 단위 프세스의 수행 리더에게 부여
	- 블랙 벡트
		- 식스 시그마의 기법을 충분히 이해하고 활용할 수 있는 핵심 인재에게 부여하는 인증 자격
		- 단위 사업부내 핵심 프로젝트 리더에게 부여
	- 마스터 블랙 벨트
		- 블랙 벨트 자격 취득 후 엄격한 심사를 거쳐 선발되는 소수의 핵심 인재에게 부여되는 인증 자격
		- 식스 시그마에 대한 교육 및 프로젝트 컨설팅을 수행하며 기업의 비전과 전략과 관련된 과제의 리더에게 부여된다.
## 기타 프로세스 표준
### ISO 12207
- 조직 또는 프로젝트 내에서 소프트웨어 수명 주기 프로세스를 정의 제어 개선 하는데 사용할 수 있는 절차를 제공한다.
- 각 프로세스 그룹에 대하여 정의 된 세부 활동들의 주체 입력정보 출력 정보 작업 내용 들을 포함하는 세부 절차를 정의
- 방대한 양의 세부 프로세스를 포함
	- 프로세스 수행에 적용할 대상을 선택 재구성하여 적용할 수 있다.

- 프로세스 그룹
	- 합의 프로세스
		- 구매자와 공급자 간의 계약 수립과 관련된 프로세스
		- 세부 프로세스로 Acquisition 프로세스와 Supply 프로세스가 있다
	- 조직 프로젝트 지원 프로세스
		- 조직에서 시스템 개발 및 관련 프로젝트를 시작하고 통제 및 관리하고 지원하기 위해서 사용될 수 있는 프로세스 영역이다.
		- 수명주기 모델 관리 프로세스, 인프라 구조 관리 프로세스, 포트폴리오 관리 프로세스, 품질 관리 프로세스등으로 구성된다.
	- 기술 관리 프로세스
		- 프로젝트계획, 프로젝트 평가, 의사결정, 리스크 관리 , 형상 관리 품질 보증과 같은 활동을 정의 하는 프로세스들로 구성된다
	- 기술 프로세스
		- 소프트웨어 개발 과직접적으로 관련된 프로셋들이다
		- 요구사항 정의, 아키택처 정의, 시스템 분석, 구현, 통합, 검증, 운영 유지등의 프로세스를 포함한다
### ISO 29110
- 최대 25명이 넘지 않는 작은 조직에서 진행하는 프로젝트를 대상으로 하는 수명 주기 프로세스를 정의
- ISO/IEC 29110에 정의된 라이프 사이클 프로세스는 소프트웨어 및 시스템 을 획득하고 사용할 뿐만 아니라 개발 및 공급할 때에도 사용 할 수 있다.

- 핵심적인 두 개의 프로세스 영역
	- 프로젝트 관리 프로세스
		- Planing
		- Execution
		- Evaluation
		- Cloure
	- 구현 프로세스
		- Initiation
		- Analysis
		- Design
		- Construction
		- Integration and test
		- Delivery
- 프로세스에서 여러 개의 세부 태스크를 정의
	- Entry
	- Intermediate
	- Advanced
	- 선택하여 세부적으로 수행할 수 있다
- 프로젝트 관리 프로세스는 117개의 태스크
- 구현 프로세슷는 164개의 태스크를 정의하고 있다
- 외부에서 제품을 구매하는 작은 기업을 위하여 획드 관리 프로세스와 이관 프로세스를 보조적으로 제시
### ISO 15288
- 사람이 만든 시스템의 수명주기를 설명하기 위한 프로세스 정의에 관한 공통 프레임워크이다
- 조직 또는 프로젝트 내에서 사용되는 시스템 수명주기 프로세스를 정의하고 프로세스 제어 및 개선을 지원하는 프로세스를 정의한단.
- 시스템 획득하고 공급할 때 요구되는 프로세스를 포함 한다.
- 주로 항공우주 시스템, 통신 시스템, 군수 시스템 드과 같이 하드웨어와 소프트웨어가 통합되는 시스템 관점의 프로세스 표준

- 프로세스 영역 구성
	- 합의 프로세스
		- 획득 프로세스와 공급 프로세스로 구성
	- 조직프로세스
		- 조직 환경 관리 프로세스, 투자 관리 프로세스, 시스템 수명주기 프로세스 관리 프로세스, 자원관리 프로세스, 품질 관리 프로세스
	- 프로젝트 프로세스
		- 프로젝트 계획, 평가, 관리, 의사결정, 위험 관리, 형상관리등과 같은 프로세스로 구성
	- 기술 프로세스
		- 요구사항 정의, 요구사항 분석, 아키텍처 설계, 구현, 통합, 검사 및 검증, 운영, 유지보수 등의 프로세스로 구성
