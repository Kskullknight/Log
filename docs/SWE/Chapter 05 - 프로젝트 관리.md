# 프로젝트 관리 필요성
- 프로젝트가 실패하는 이유 
	- 소프트웨어 마인드가 부족하다
		- 소프트웨어는 물리적으로 존재하지 않기 때문에 언제든지 변경 가능하다고 생각한다.
	- 적절한 소프트웨어 공학 기술을 활용하지 못하고 있다
		- 체계성과 효율성을 지원하는 다양한 기법이 있지만 업무 증가 일정 징녀등의 이유로 공학적 기법 적용을 미룬다
	- 적절한 프로젝트 관리 기술이 부족하다
		- 개발 기술이 고도화되고 응용 도메인이 다양해지면서 고려해야할 요소가 기하 급수적으로 늘어난다.
- 프로젝트가 실패했다는 이유는 단순히 정확한 기능의 소프트웨어를 개발했다는 것만 의미하지 않는다
	- 기간 초과, 예산 초과 등의 문제가 발생했다면 프로젝트는 실패했다고 볼 수 있다.
## 프로젝트 관리의 정의
- 개발 목표를 달성하기 위해서 요구되는 제반 장비와 환경ㅇ르 마련하고 이를 일정하게 유지해주는 것
## 프로젝트 관리 단계
### 1. 계획 수립
- 정확한 계획을 수립
	- 소프트웨어 개발의 목적
	- 필요한 자원
	- 정보의 흐름
	- 소요 인력
	- 산출물
	- 개발에 필요한 기간
	- 인력의 수
	- 필요한 장비와 시설
	- 각 시점에 필요한 산출물
	- 교육의 필요성
### 2. 자원 획득
- 프로젝트가 수행될 여건 마련
	- 예측한 다양한 장비 및 시설 확보
	- 팀구성 (R&R)
### 3. 실행
- 정의한 일정에 따라 소프트웨어 개발 활동을 수행
- 일정계획에 따른 진도 관리
- 변경 관리 및 품질 관리 활동
- 리스크 관리
### 4. 모니터링
- 베이스 라인에 근거하여 개발 활동에 대한 진척도를 관리하고 품질을 모니터링
- 설정된 베이스 라인마다 단계별 산출물을 검토하여 소프트웨어가 올바르게 개발되고 있는지 확인한다
- 사용자의 요구사항을 만족하는지 확인
- 편차가 발생한는 경우 편차 발생의 원인을 분석하고 적절하게 대응한다.
### 가장 중요한 요소
- 좋은 사람을 구하는 것
	- 개발 기술이 우수하고 경험이 풍부한 사람
## 프로젝트 관리의 실패의 원인
- 부정확한 요구사항
	- 소프트웨어 개발 = 사용자의 요구사항을 가감 없이 프로그램으로 변환하는 과정
	- 사용자의 요구사항이 부정확하다면 부정확한 소프트웨어 개발을 하게 된다
- 사용자 환경 이해 부족
	- 사용자의 비즈니스 전략이나 정책 변화를 이해해야 한다.
	- 개발의 궁극적인 필요성을 이해야 한다
- 부정확한 프로젝트 계획
	- 아직 일어나지 않은 일들을 예측하는 것
	- 정확한 일정 계획, 리스크 예측등을 통해 계획 대비 편차를 줄여야 한다.
- 부족한 자원
	- 인력, 시간, 예산등이 정확히 산정되어야 한다
	- RFP에서 제시하는 요구사항 대비 사업 예산이 부족하지 않도록 산정해야 한다.
- 높은 사용자 기대치
	- 유사한 다른 시스템보다 우수하고 보기 좋은 시스넴이 개발되기를 원한다
	- 불필요한기능의 개발은 리스크 증가 및 개발 기간 연장을 초래한다
- 변경 관리 부족
	- 변경은 피할 수 없음
		- 요구사항, 설계등
	- 변경이 전체 프로젝트에 일관성있게 관리되괴 반영되어야 하는데 일부만 변경되는 경우
- 자원 관리 실패
	- 계획 수립 과정에서 산정된 자원을 적절하게 분배하고 활용해야 한다.
	- 프로젝트 관리 대비 편차가 발생하는 경우 자원관리에 실패할 가능성이 높아진다.
- 개발자 기술 부족
	- 개발자가 최신 기술을 확보하지 못했다면 개발 기간 연장, 개발 기능의 완성도 부족등의 문제 발생
- 프로젝트 관리 기술 부족
	- 관리자의 경험과 지식을 바탕으로 문제상황에 대한 처방전이 나옴
		- 계획과 편차가 발생했을 때
		- 예상치 못한 리스크가 발생했을 때
# 프로젝트 관리 기법
- 소프트웨어는 눈에 보이지 않고 만질 수도 없다
	- 아직 수행하지 않은 일들을 예상하고  계획하기 쉽지 않다
- 개발자의 수행 역량을정확히 알 수 없음으로 정확한 산정이 불가능하다
	- 예측된 값을 기반으로 필요한 인력의 규모를 산정해야 한다.
## 일정 관리 기법
- 일정 관리느 가장 기본적인 사항이다
- 일정은 다양한 요소를 고려하여 계획해야 한다
	- 요구사항의 복잡도
	- 개발자 규모 및 기술 수준
	- 팀 구성 형태
	- 개발 중에 발생 가능한 리스크 정도
- 일정 계획에서 가장 기본이 되는 입력 사항
	- 프로세스 모델
		- 어떤 프로세스 모델을 선택하는 지에 따라 개발 단계의 구성에 차이가 생긴다
	- 베이스 라인 정의
		- 변경이 발생했을 때 변경 사항을 결정한 다은 후속 작업을 수행하기 위한 출발점으로 정의
### 베이스라인 관리
- 베이스라인이 많은 경우 빈번한 회의와 의사결정과정을 걸처야하기 때문에 시간과 노력이 낭비된다.
	- 너무 적은 경우 의사결정이 이루어지지 못하는 경우 발생하여 재작이 필요해짐
- 적절한 베이스 라인 선정이 중요하다
	- 요구사항 베이스라인
		- 확정된 요구사항 정의서가 다음 작업의 기준 정보로 사용될 수 있다
	- 기능 베이스라인
		- 사용자 요구사항을 시스템의 기능으로 할당한 후에 작성하는 요구사항 명세서와 추적성 메트릭스를 기반으로 향후 작업의 기준점으로 설정
	- 제품 베이스 라인
		- 개발 완료후 정의하는 베이스 라인
		- 개발 과정의 모든 산출물이 해당 베이스라인의 기준선에 포함됨
### 작업 분할도 (WBS, Work Breakdown Structure)
- 프로젝트의 전체 목표를 준간의 세부 목표들로 쪼개어 나타낸 작업 목록이다
- 작성하는 이유
	- 프로젝트에서 수행해야 하는 모든 종류의 작업을 식별하기 위해
- 태스크는 추정 가능한 양의 작업이며, 독립적으로 이루어 질 수 있을 때까지 분할 해야 한다.
- 태스크 단위의 식별자를 기준으로
	- 수행 시간
	- 선행 / 후행 태스크
	- 태스크 담당자 정보
- 일정 요약표에 해당하며 구체적인 스케줄링을 위한 입력으로 사용된다.
- 프로젝트 전체 소요 시간에 대한 정보를 알 수 있다
- 유의할 점
	- 반드시 수행해야 할 작업을 작업 분할도의 태스크로 식별하지 못하면 일정 지연이 발생할 수 있다.
		- 문서화 작업, 프로젝트 리뷰를 통해서 태스크를 식별해야 한다
	- 각 태스크 처리 시간을 산정할 때 리뷰 활동을 반드시 고려해야 한다.
### 간트 차트
- 프로젝트의 스케줄링, 예산 산정, 자원 계획을 수립하기 위해 사용하는 일정 표현 기법
- 일반적인 바 차트 형태로 표시하며 바의 길이에 비례하여 소요 시간을 나타낸다
- 슬랙 타임
	- 작업을 완료해야 하는 마지막 시간

### 퍼트(PERT) 차트
- Program Evaluation and Review Techniques
- 프로젝트에서 수행되어야 하는 태스크 간의 의존성 관게를 표현하는 차트
- 원(혹은 박스)과 화살표를 이용하여 나타냄
	- 원은 태스트
		- 완료하는데 드는 일장
	- 화살표는 태스크 간의 의존성
		- 병렬 실행과 의존성을 표시
		- 태스크 수해 일수의 합이 가장 많은 경로가 임계 경로가 된다.
- 프로젝트 전체 수행 일정에서 핵심적인 수행 결로를 식별하고 이를 관리할 수 있다.
- 임계 경로의 태스크에 지연을 발생하는 경우 전체 일정의 지연이 발생한다
## 비용 관리 기법
- 정확한 개발 기간와 비용은 에측하기가 매우 어려움
	- 최대한 근사치를 예측해야 한다
### COCOMO / COCOMO II
- Constructive Cost Model의 약자
	- 베리 보햄이 개발한 비용 산정 모델
- KDSI(Kilo Delivered Source Instructions)을 기반으로 비용을 산정
> [!note] KDSI란?
>소프트웨어 규모를 측정하는 단위: 납품된 소스 코드 1,000줄(line)을 의미

- 개발 대상에 대한 KDSO를 유사 소프트웨어를 비교 분석하여 산출후, 복잡도에 근거하여 세 가지 모델중 하나을 선택하여 개발 긴간과 필요한 인원수를 산정

| 개발 모드                | 소요 노력                     | 소요일                        |
| -------------------- | ------------------------- | -------------------------- |
| Organic (조직형)        | $E = 2.4 × (KDSI)^{1.05}$ | $D = 2.5 × (PMDEV)^{0.38}$ |
| Semi-detached (반결합형) | $E = 3.0 × (KDSI)^{1.12}$ | $D = 2.5 × (PMDEV)^{0.35}$ |
| Embedded (내장형)       | $E = 3.6 × (KDSI)^{1.20}$ | $D = 2.5 × (PMDEV)^{0.32}$ |
- PM = Person Month
- PMDEV = PM X Effort Multipier
- TDV: 프로젝트 종료하는데 소요되는 개월 수

- Effort Multipier: 노력 승수
	- 제품 속성
	- 컴퓨터 속성
	- 개발자 속성
	- 프로젝트 속성으로 그룹핑된 15가지 비용 유발 인자를 정의
	- 각 인자별로 0.7에서 1.6까지 가중치를 갖는 스케일을 정의
	- 15가지 인자에 대한 값을 선정하고 모두 곱함녀 최종 노력 승수 값을 얻을 수 있다.
	- 값이 클수록 소프트웨어의 운영 환경이나 개발 과정이 복잡해짐
	- 전체 소요 일정 및 노력이 증가하도록 설계

#### COCOMO II
- 단순한 소스코드의 랑니 수만이 아니라 소프트웨어 기능의 양과 복잡도를 기준을 규모를 산정하는 기능 점수 분석 깁버을 혼합하여 비용을 산정하도록

### 전문가 판단
- 전문가 판단에 따라 소프트웨어 개발 비용 혹은 소프트웨어 규모를 산정
	- 다수의 전문가가 모여 요구사항을 기준으로 소프트웨어 크기를 예상
	- 반복적 전문가 평가를 통해 소프트웨어 규모를 추정한다.

- 규모 산정 절차
	- Workshop
		- 전문가 그룹을 선정하고 WBD 진행 방식을 설명
	- Experts Initialization
		- 전문가 그룹에 의사 결정이 필요한 정보를 제공한다
	- Kickoff meeting
		- 전문가들이 의사결정이 필요한 내용에 대해 논의 한다.
	- Individual Estimation
		- 전문가들이 자신의 의견을 정량화 해서 규모를 산정한다.
	- Tabulating
		- 각 전문가가 작성한 결과를 취합하여, 익명의 결과로 전환한 통합 버전을 작성한 후에 모든 전문가에게 피드백한다.
	- Reviewing & Consensus
		- 다른 전문가들의 의견과 자신의 의견에 나타난 차이를 확인 후, 다시 전문가들이 재논의를 수행한다.
		- 규모 산정에 추가로 고려할 이슈가 있다면 개별 산정 단계에서 반복적으로 평가를 실시한다.
		- 의견일치가 이루어 지면 최종적으로 소프트웨어 규모 확정을 종료

### 파킨슨의 법칙
- 소프트웨어 개발 비용이나 필요한 인력 수는 소프트웨어 복잡도및 기능 수와 관련
- 최종적으로는 개발 조직에서 가용한 예산과 인력에 의존하여 결정된다고 주장
- 어떤 방법으로 개발 비용을 산정하더라도 
	- 조직에서 확보한 인력이 해당 소프트웨어를 개발하기에 충부한지 여부와 무관하게 
	- 조직의 인력에 개발 일정이 맞춰져 진행된다는 의미
- 자체 개발 방식으로 진행하는 경우나 적은 규모의 소프트웨어 개발 조직에는 적합 할 수 있다
### 기능 점수 산정법
- 기능 점수를 사용해 소프트웨어 비용을 산정하는 방법
- 최근에 표준 비용 산정 방법으로 활용
- 소프트웨어 시스템이 제공하는 기능의 필요 정도와 이의 복잡도를 기준으로 평가하는 방법

### 그외
- 동일한 도메인의 유사 프로젝트를 분석하여 소프트ㅜ에어의 규모를 산정 하는 방법
- 가장 작은 단위의 태스크에 대한 개발 비용을 산정한 후 이들을 바텀업으로 누적하여 산정하는 방법
	
## 위험 관리
- 개발 과정에서 리스크가 발생하면 프로젝트느 상당히 지연되거나 실패로 이어질 수 있다.
- 리스크를 사전에 예측하고 이를 에방 또는 완화하기 위한 계획을 수립해야 한다.
- 전형적인 리스크 유형
	- 요구사항의 빈번한 변경

### 프로젝트 리더

- 팀 간의 기술적 조율
- 일정 조정 및 모니터링
- 프로젝트 문서 작성및 통합
### 프로젝트 팀장

- 일반적으로 TL과 PL과 능력이 동일한 경우 많음
### 프로그램 엔지니어
- 넓은 의미에서 엔지니어, 좁은 의미에서 개발자

	- 분석 설계자
	- 프로그래머
	- 데이터베이스 개발자
	- UI개발자
	- 등등
### 형상 관리자

### 품질 관리자
- 산출물에 대한 오류 확인과 결함 검출을 위한 리뷰, 테스트 활동 계획및 추진
- 전사차원에서 지원하는 조직
### 리어종 그룹

- 대형 프로젝트나 전문성이 높은 프로젝트에서 운영

## 프로젝트 팀 구조
- 의사소통 경로를 중심으로 구분
### 중앙집중형 팀 구조
- 작업 목록이 단순하거나 충분히 이해된 경우에 적합한 팀 구성 방식
- 프로젝트 관리자나 리더가 프로젝트와 관련도니 기술 내용ㅇ과 시스템 설계에 대한 상세 정보를 잘 알고 있어야함
	- 따라서 유능한 리더가 필요
	- 리더에게 업무의 오버로드가 걸릴 수 있다.
- 문제 해결이 신속하게 이루어짐
- 의사소통 패턴이 매우 단순함
> 짧은 기간 정확한 업무를 신속하게 처리하는데 적합하다.
### 분산형 팀 구조
- 주요 의사결정이 팀 구성원의 합의에 의해 이루어지는 민주주의적 팀 구성
- 모든 팀 구성원이 다른 사람의 작업 수행 결과를 공유하면서 프로젝트의 발생 가능한 문제를 해결
- 의견을 수렴하는 의사소통 패턴
	- 의사소통 패턴이 복잡함으로 대규모 구성원을 포함하는 프로젝트에는 적합하지 않다
- 팀원 구성원 간의 사기를 높이고 팀원 교체를 줄일 수 있다.
- 작업 내용을 충분히 이해하기 못한 경우, 문제가 복잡하여 해결 방안을 모색해야 하는 경우
- 장기간에 걸쳐 수행하는 프로젝에 적용
### 하이브리드 팀 구조
- 중앙집중형 팀 구조와 분산형 팀 구조를 통합한 계층형 팀 구조
- 두 가지 팀 구조의 장점을 도입하기 위해 구성됨
- 일반적으로 많은 프로젝트에서 적용하는 팀 구성방식
- 프로젝트 관리자는 각 팀의 리더와 중요한 의사결정을 하기 위한 중앙 집중형 구조
	- 하지만 팀 내부의 운영은 부산형 구조를 채택하여 의사소통
- 팀원을 소규모로 구성하기 때문에 팀별 문제 해결이 용이하다.
- 팀 간 문제는 프로젝트 관리자가 조정한다.
### 팀 구조 산정 전략
- 프로젝트 기간, 문제의 복잡성, 참여 인력 등을 고려하여 팀을 구성
- 모든 프로젝트에 적합한 팀 구성은 없다.
	- 분산형 팀 구조 - 엔지니어들간의 의사소통이 필요한 경우
	- 중앙집중형 팀 구조 - 신속한 개발
#### 팀 구성시 고려 사항
- 비용 절감 측정에서 효과적인가
- 팀 구성원의 교체발생 가능성
- 신입 엔지니어에게 프로젝트에 대한 이해나 경험을 높일 수 있는 기회를 제공 가능성
- 최신 기술 또는 특정 지식에 대한 공유및 전파 가능성
## 전사적 운영 조직
- 매트릭스 기반의 프로젝트 팀 구성
	- 모든 엔지니어는 필요시 해당 프로젝트에 참여하여 자신의 업무를 수행한 후 다른 팀에 참여하기 위해 해당 팀을 나온다
	- 전사 차원의 프로젝트에서 적용
### 품질관리 팀
- 전사 차원에서 운영되는 대표적인 팀
- 산풀뭉에 대한 리뷰, 인스펙션, 테스트등과 같은 활동을 지원
- 프로젝트 수행 경험이 있고 전문적인 지식을 확보한 중급 이상의 엔지니어들로 구성
- 전사차원의 품질 관리 가이드 라인을 마련
- 품질관리를 위한 지원 도구 관리 및 품질관리 활동에 필요한 지식 교육 담당
- 필요에 따라서 형상 변경 및 관리 활동 수행
### PMO (Project Management Office)
- 전사 차원에서 프로젝트 생성, 진행 과정 모니터링, 프로젝트 단위 리스크 해결 지원
- 프로젝트 진행의 효율성을 높임
- 종료된 프로젝트, 진행중인 프로젝트, 게획중인 프로젝트등에 대한 포트폴리오 관리
- 단위 프로젝트 관리자를 지원하고 통제

- 프로젝트가 대형화, 복잡화되면서 프로젝트를 성공시키기 위해 조직 차원에서 엔지니어의 역량을 어디에 집중 혹은 분배
- 비즈니스 영역의 사업 우선순위를 위한 분석및 결정
- 크게 기술적인 분야과 관리적 분야로 나뉜다.
	- 기술적 분야 - 프로젝트 총괄 관리자, 품질 관리자, 형상 관리자, 기술 분야 전문가등으로 구성
	- 관리 분야 - 고객 담당자, 재무 관리자, 마케 플래너 등으로 구성

## 프로젝트 관리 기능
- 프로젝트의 전체 수명 주기를 관리
### 계획 기능
- 프로젝트를 수행할 구성원을 등록하고 팀을 구성한다
- WBS를 생성하고 팀원에게 할당
- 태스크별 일정 계획을 생서하고 검토및 조정
- 프로젝트 수행의 주요한 마일스톤을 정의한다.
### 진도 관리 기능
- 각 팀원은 자신에게 할당된 태스크를 갭려적으로 수행하고 수행 완료 및 진도 정보를 입력한다
- 테스크 수행과 관련된 리스크가 발생하면 이슈를 제기
### 협업 기능
- 팀원 혹은 팀 간 협업이 요구된는 태스크에 대해서 협업 목표를 설정한다
- 협업 태스크를 수행하기 위한 협업 공간을 생성하고, 협업자를 참여 시킨다.
- 상호 의사소통과 자료 공유를 위한 기반을 제공한다.
### 배포 기능
- 태스크 수행이 완료되면 산출물을 저장소에 저장한다
- 산출물 저장을 위해 check-in / check-out 기능을 제공한다.
- 자료에 대한 신속한 검색 기능을 제공한다.
### 보고서 생성 기능
- 다양한 보고서 생성이 가능하다
- 팀원을 위한 태스크 목록, 작업 일정 보고서가 생성된다
- 관리자를 위한 조합 보고서가 제공된다
- 일정 보고서, 리스크 및 이슈 보고서, 진도 보고서등이 생성된다
- 그래프, 파이 차트 등과 같은 형태의 보고서 생성이 가능한다.
### 통합 가능
- 프로젝트 차원에서 활용 가능한 기능을 포함한다
- 프로젝트 수행에 딜요한 템플릭을 참조 모델로 제공한다
- 일정 달력, 드롭 박스등의 기능이 제공된다
## PMO 도구
- 전사 차원에서 소프트웨어 프로젝트를 모니터링하고 통데, 관리하기 위한 PMO조직은 소프트웨어 비즈니스를 위한 필수 조직이다
- PMO의 역할을 기반으로 한 주요 기능
### 프로젝트 총합 관리 기능
- 프로젝트 진행의 적정성을 검초하기 위한 Feasibility분석이 제공된다
- 프로젝트 수행에 대한 계획 대비 실적을 모니터링 한다
- 단위 프로젝트의 요청사항을 수렴하고 대응한다
- 프로젝트 수행에 대한 실적을 분서하고 보고한다
- 프로젝트의 목표 달성을 보장하기 위해 일정, 범위 등에 대한 관리 지침을 개발한다.
### 프로젝트 범위 관리 기능
- 프롲게트에서 정의된 범위 대비 수행 결과를 비교,분석한다.
- 프롲게트 수행 범위에 대한 중복성 확인을 통해 조정 및 통제한다
- WBS 생성 기반으로 프로젝트 범위의 타당성을 검토한다
- 프로젝트 수행 계획서에 근거한 수행 결과를 확인 ,검증한다.
- 프로젝트 범위에 대한 변경 및 변경내역을 관리한다.
### 일정 및 진도 관리 기능
- 프로젝트 일정 계획의 조정 및 계획 대비 진도 관리를 수행한다
- WBS의 태스크 단위 일정에 대한 타당성을 검토하고 조정한다
- 태스크 단위 수행에 대한 프로젝트 전체 차원의 진도를 관리한다
### 산출물 품질 관리 기능
- 요구사항 및 품질 요건을 만족하기 위한 지속적인 품질 평가를 지원한다
- 산출물별 적정성 및 산출물 간 일관성을 평가한다
- 산출물 작성 가이드 라인, 방법론, 표준 준수성등을 평가한다
### 인적,물적 자원 관리 기능
- 단위 프로젝트 투입 린려 현황을 관리한다
- 투입 인력에 대한 적정성 및 변화 인력 영샹도를 검토한다
- 프로젝트에서 요구하는 자원의 현황, 운영 ,관리의 적정성을 검토한다
### 이슈 및 위험 관리 기능
- 전사 차원에서 이슈 및 위험 관리 체계를 정의하고 공지한다
- 방생한 이슈/위험에 대안 현황 파악, 대응 방안, 해결 방안, 처리 결과등을 관리한다
- 이슈 및 위험에 대한 보고 체계를 구성한다
- 전체적인 이슈 및 위험에 대한 유형 분류, 해결 수준등을 분석한다.
### 의사소통 관리 기능
- 프로젝트 관련 이해관계자간의 의사소통 경로를 지원한다
- 의사소토 결과(회의록 등)를 관리하고 보고 체계를 운영한다
- 결정된 사항에 대한 터리 ㅁ치 후속 사항에 대하여 추적, 모니터링 한다.
### 성과 관리 기능
- 프롲게트에서 발생한 겅과를 관리, 활용하기 위한 계획을 수립한다
- 프로젝트 성과 식별을 위한 지표를 정의하고 제시한다.
- 프로젝트의 핵심 성과 지표를 측정하고 측정 결과를 보고한다.
### 조직의 능력 관리 기능
- 프로젝트 관리, 운영에 필요한 교육 훈련을 실시한다
- 신규 비즈니스에 적합한 프로젝트 가이드 라인을 변경혹은 개발한다.
- 프롲게트 결과물에 대한 정보 보안 대책을 수립한다.
## 엔지니어링 도구
- 소프트웨어 분석, 설계,코딩, 테스트를 지원하는 데 사용되는 도구
- UML 기반의 객체지향 분석 및 설계 도구, 코딩 도구, 테스트 도구등을 포함한다
- 매우 다양한 도구가 상요 혹은 프리웨어로 제공
	- 이러한 도구들과 연동된 무선화 도구도 쉽게 찾아 볼 수 있다
- 소프트웨어 분석 및 설계 과정에서 엔지니어의 실수 즉 오류를 줄이는 좋은 수단
- 협업 수행을 쉽게 할 수 있도록 지원
